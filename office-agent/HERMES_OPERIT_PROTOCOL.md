# Office Agent Hermes / Operit Protocol

This document defines the stable contract for calling Office Agent from Operit, Hermes, OpenCode, or another upper-layer Agent.

## Core Rule

Office Agent tasks are two-stage by default:

```text
plan -> user confirmation -> run -> poll task -> deliver artifacts
```

High-risk and medium-risk tasks must not skip the confirmation gate. The service rejects `/office/run` unless `confirm=true` is supplied.

## Task Model

A task has these common fields:

| Field | Meaning |
|---|---|
| `task_id` | Stable ID for polling and audit |
| `task_type` | `generate`, `convert`, or `modify` |
| `status` | `planned`, `queued`, `running`, `succeeded`, `failed`, or `cancelled` |
| `workspace` | Directory containing task artifacts |
| `events` | JSONL event log path for audit and debugging |
| `plan_json` | Machine-readable plan |
| `plan_md` | Human-readable plan shown to user |
| `output` | Main generated or modified file |
| `quality_report` | Structured quality JSON |
| `quality_markdown` | Human-readable quality report |
| `requires_user_confirmation` | Whether upper layer must ask the user |
| `confirmed` | Whether execution has been explicitly confirmed |
| `result` | Final artifact paths after execution |
| `error` | Structured failure info |

## HTTP API

Start service from WSL:

```bash
cd /mnt/f/scripts/office-agent
python3 office_service.py --host 127.0.0.1 --port 8765
```

Health check:

```http
GET /health
```

Create plan:

```http
POST /office/plan
Content-Type: application/json
```

Run after confirmation:

```http
POST /office/run
Content-Type: application/json

{
  "task_id": "...",
  "confirm": true
}
```

Poll status:

```http
GET /office/tasks/{task_id}
```

Read event log:

```http
GET /office/tasks/{task_id}/events
```

Cancel before or during execution:

```http
POST /office/cancel
Content-Type: application/json

{
  "task_id": "..."
}
```

Equivalent task-scoped cancel endpoint:

```http
POST /office/tasks/{task_id}/cancel
```

## Generate Request

```json
{
  "task_type": "generate",
  "kind": "docx",
  "spec": {
    "title": "Project Brief",
    "sections": [
      {"heading": "Overview", "paragraphs": ["Generated by Office Agent."]}
    ]
  }
}
```

`kind` can be `docx`, `pptx`, or `xlsx`. The caller may also pass `spec_path` instead of inline `spec`.

## Convert Request

```json
{
  "task_type": "convert",
  "conversion": "pptx-to-docx",
  "input": "/mnt/f/scripts/office-agent/examples/regression_inputs/high_density.pptx",
  "fidelity": "F2",
  "include_images": false
}
```

## Modify Request

```json
{
  "task_type": "modify",
  "input": "/mnt/f/office-output/agent-validation/generate-docx/example.docx",
  "instruction": {
    "operations": [
      {"op": "append_section", "heading": "Next Steps", "paragraphs": ["Review and approve."]}
    ]
  }
}
```

The caller may also pass `instruction_path` instead of inline `instruction`. Supported modify operations include `replace_text`, `append_section`, `add_toc`, `set_heading_style`, `append_slide`, `set_theme`, `set_cell`, `insert_formula`, `append_rows`, and `style_header`.

## Hermes Responsibilities

Hermes should:

1. Create a task with `/office/plan`.
2. Store `task_id`, `plan_json`, `plan_md`, and `workspace`.
3. Return `plan_md` to Operit for user review.
4. Wait for explicit user confirmation.
5. Call `/office/run` with `confirm=true`.
6. Poll `/office/tasks/{task_id}` until status is `succeeded`, `failed`, or `cancelled`.
7. Read `/office/tasks/{task_id}/events` when debugging or writing an audit trail.
8. Return `output`, `quality_report`, `quality_markdown`, and any ledger/change-log artifacts.
9. Write important decisions and artifact paths to Obsidian.

## Operit Responsibilities

Operit should:

1. Collect user intent, source files, and output preference.
2. Display `plan_md` before execution.
3. Ask the user to confirm mode, fidelity, image handling, template, and output path.
4. Never auto-confirm high-risk materials.
5. Display result paths and quality status after completion.
6. If the user cancels before execution, call `/office/cancel` or `/office/tasks/{task_id}/cancel`.

## Error Handling

HTTP errors use this shape:

```json
{
  "ok": false,
  "error": {
    "code": "bad_request",
    "message": "task_id is required",
    "details": {"type": "ValueError"}
  }
}
```

Task failures are stored in the task record:

```json
{
  "status": "failed",
  "error": {
    "type": "ValueError",
    "message": "..."
  }
}
```

## MCP-Style Stdio Bridge

Start bridge:

```bash
cd /mnt/f/scripts/office-agent
python3 office_mcp_server.py
```

Tools:

| Tool | Purpose |
|---|---|
| `office_plan` | Create plan and task record |
| `office_run` | Run after explicit confirmation |
| `office_get_task` | Read task status and artifact paths |
| `office_get_events` | Read task event log |
| `office_cancel` | Request cancellation |

The bridge uses JSON-RPC lines over stdin/stdout and is intentionally dependency-free. A future version can replace it with a full MCP SDK server without changing the high-level tool contract.
